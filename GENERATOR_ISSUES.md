# Generator Issues

Tracker for generator behavior issues that need design discussion or implementation. Keep entries evidence-based and update status as work lands.

## Status Legend

- `Open`: confirmed issue or risk, no implementation started.
- `Design`: solution direction being discussed.
- `In Progress`: code/doc changes underway.
- `Done`: implemented and verified.
- `Deferred`: valid issue, intentionally postponed.

## G-001: TEMP Marker Address Collisions

Status: Open

### Symptom

Generated PLC tags in the `TEMP` tag table reuse fixed marker address ranges every time generation runs. If only a new unit is generated later, addresses can overlap with tags from previously generated units and must be resolved manually in TIA Portal.

### Evidence

- `TIAOpenness\TIA_Lib\TIA_LIB\PlcProject.cs`
- `CreateTags()` finds or creates the `TEMP` PLC tag table.
- Fixed starting ranges are hardcoded:
  - valves: `M500.x`
  - digital inputs: `M1000.x`
  - control valves: `MW1500`
  - analog inputs: `MW2000`
  - motors: `M2500.x`

### Proposed Direction

Introduce a marker address allocator that scans existing PLC tag addresses before creating new tags, then assigns the next free address in the appropriate preferred range.

The allocator should consider marker memory globally, not just inside `TEMP`, because `M`, `MB`, `MW`, and `MD` addresses overlap across all PLC tag tables.

### Verification Ideas

- Existing `TEMP` tags at the preferred base addresses are skipped.
- Existing tags in another PLC tag table are also treated as occupied.
- Generated bool tags do not overlap generated word tags.
- Re-running generation does not move existing generated tags unnecessarily.

## G-002: Global PLC Tag Name Collisions Across Units

Status: Open

### Symptom

Generated external PLC tag names are based on the device name only. Two units with the same device name, for example `Q21`, can produce the same global tags such as `CTRL_Q21`, `FB_OPN_Q21`, or `IN_Q21`.

### Evidence

- `TIAOpenness\TIA_Lib\TIA_LIB\PlcProject.cs`
- `AddAnalog(...)` registers `IN_` + device name.
- `AddDigital(...)` registers `IN_` + device name.
- `AddValve(...)` registers `FB_OPN_`, `FB_CLS_`, and `CTRL_` + device name.
- Device constructors wire those same bare names into generated networks:
  - `Devices\Analog.cs`: `IN_` + tag name to `fbMonAn.IN_INT`
  - `Devices\Digital.cs`: `IN_` + tag name and `_QB` to `fbMonDi`
  - `Devices\Valve.cs`: `FB_OPN_`, `FB_CLS_`, and `CTRL_` + tag name
  - `Devices\Motor.cs`: `FB_ON_` and `CTRL_` + tag name
  - `Devices\ValveControl.cs`: `CTRL_` + tag name

### Proposed Direction

Keep the internal unit device name unchanged, but make generated external PLC tag names unit-scoped.

Example direction:

```text
BL170_Q21_CTRL
BL170_Q21_FB_OPN
BL170_Q21_FB_CLS
BL170_Bl171_B13_IN
BL170_Bl171_B13_IN_QB
BL170_QC_B24_IN
```

Use one helper for all generated external tag names so `CreateTags()` and network wiring cannot drift apart.

### Verification Ideas

- Two units with a `Q21` device generate distinct TEMP tags.
- Each generated block network points to the unit-scoped tag names.
- Existing internal network names, local instance names, config names, and HMI object names are not changed unless explicitly intended.

## G-003: Missing Generated PLC Tag Comments

Status: Open

### Symptom

Generated TEMP tags have no generated comments, making manual review and maintenance in TIA Portal harder.

### Evidence

- `TIAOpenness\TIA_Lib\TIA_LIB\PlcProject.cs`
- `CreateTags()` calls `table.Tags.Create(...)` without setting comments or descriptions.

### Proposed Direction

After creating a tag, set a concise generated comment containing at least:

- generator name
- unit name
- original device name
- role
- optional legacy/generated source name during migration

Example:

```text
Generated by etu-process-generator-TiAv20; Unit BL170; Device Q21; Role valve command
```

### Verification Ideas

- Newly created TEMP tags show the expected comment in TIA Portal.
- Existing user comments are not overwritten without an explicit migration decision.

## G-004: Bit Allocator Appears To Skip Bit 7

Status: Open

### Symptom

Bool marker allocation increments `countBit`, then advances the byte when `countBit == 7`. That appears to skip addresses like `M500.7`, `M1000.7`, and `M2500.7`.

### Evidence

- `TIAOpenness\TIA_Lib\TIA_LIB\PlcProject.cs`
- Repeated pattern:

```csharp
table.Tags.Create(..., "M" + countByte + "." + countBit++);
if (countBit == 7)
{
    countByte++;
    countBit = 0;
}
```

### Proposed Direction

If retaining sequential bit allocation, advance after bit 7 has been used. A central allocator would remove this repeated pattern and reduce off-by-one risk.

### Verification Ideas

- Eight bool tags allocate bits `.0` through `.7` before moving to the next byte.
- Allocation still avoids occupied bits discovered from existing tags.

## G-005: Existing Tag With Wrong Type Or Address Is Silently Accepted

Status: Open

### Symptom

`CreateTags()` checks only whether a tag with the expected name exists. If the existing tag has the wrong data type, wrong address, or stale comment, generation does not report or repair it.

### Evidence

- `TIAOpenness\TIA_Lib\TIA_LIB\PlcProject.cs`
- Repeated pattern:

```csharp
if (table.Tags.Find(name) == null)
{
    table.Tags.Create(name, type, address);
}
```

### Proposed Direction

When an expected generated tag already exists, validate at least type and logical address. Decide whether to report only, repair generated-owned tags, or fail generation.

### Verification Ideas

- A matching existing tag is left unchanged.
- A mismatched existing tag is reported clearly.
- User-owned tags are not overwritten by default.

## G-006: Motor Feedback Tag Creation Depends On CTRL Missing

Status: Open

### Symptom

Motor `FB_ON` tag creation happens inside the branch that only runs when `CTRL` is missing. If `CTRL_*` exists but `FB_ON_*` does not, the feedback tag may not be created.

### Evidence

- `TIAOpenness\TIA_Lib\TIA_LIB\PlcProject.cs`
- In the `Motors` loop, `table.Tags.Create(motor.FB_ON, ...)` is nested inside `if (table.Tags.Find(motor.CTRL) == null)`.

### Proposed Direction

Create or validate `CTRL` and `FB_ON` independently.

### Verification Ideas

- Existing `CTRL_*` plus missing `FB_ON_*` results in only the missing feedback tag being created.
- Existing feedback tag is not duplicated.

## G-007: HMI And SiVArc Impact Of Unit-Scoped External Tags

Status: Open

### Symptom

Unit-scoping generated PLC tag names is likely necessary, but the generator also feeds SiVArc and Unified HMI workflows. Changing names in PLC network wiring may affect generated HMI tags, faceplates, or tag cleanup behavior.

### Evidence

- `Project.cs` generation flow calls:
  - `CreateTags()`
  - `Upload()`
  - `Portal.CheckTags()`
  - `GenerateSiVarc()`
  - `Portal.DeleteRenamedTags()`
  - `UserInterface()`
- `TIAOpenness\TIA_Lib\TIA_LIB\SiemensPortal.cs` contains HMI tag creation and cleanup behavior.

### Proposed Direction

Before implementing unit-scoped external tags broadly, trace one simple analog and one simple valve through generated PLC XML, HMI tag generation, and `CheckTags()` expectations.

### Verification Ideas

- SiVArc generation still creates expected HMI tags.
- `CheckTags()` does not delete valid unit-scoped generated tags.
- Unified faceplate properties still resolve the expected HMI tags and config data.

## G-008: Consider A Generated DB Instead Of Marker Memory

Status: Deferred

### Symptom

Marker memory is global, address-sensitive, and easy to collide with manual project content.

### Proposed Direction

For a larger future change, consider replacing generated TEMP marker tags with a generated DB structure for external signal staging, if TIA Portal, SiVArc, and existing PLC patterns allow it.

This is intentionally deferred because it would be a broader architectural change than fixing the current TEMP tag generator.

## G-009: External Project Definition Template

Status: Design

### Symptom

Creating or changing a unit currently requires editing internal C# code in `Project.cs`, recompiling, and treating process engineering data as application source. That makes each new unit a developer workflow instead of a portable generator input workflow.

### Evidence

- `etu-process-generator-TiAv20\Project.cs` contains active hardcoded calls such as `AddAnalog(...)`, `AddDigital(...)`, and `AddValve(...)`.
- `GENERATOR_SNIPPETS_KNOWHOW.md` already describes engineering-data-to-`Add...` call preparation, which implies the source data can be represented as structured rows.
- The app already references `GemBox.Spreadsheet`, so reading Excel workbooks is available without adding a new dependency.

### Proposed Direction

Move unit and component definitions out of `Project.cs` into a versioned external project-definition template, preferably `.xlsx` for the first implementation.

`Project.cs` should become orchestration code:

```text
Project definition workbook
        -> ProjectDefinition model
        -> validation and normalization
        -> AddAnalog/AddDigital/AddValve/etc.
        -> CreateTags/Upload/SiVArc/UserInterface flow
```

The generator should interpret the workbook directly. Avoid generating C# from Excel as the primary workflow, because that still leaves users reviewing and compiling generated source code.

### Initial Workbook Proposal

Use a workbook with a clear template version and separate sheets by data shape.

#### Sheet: `Metadata`

| Column | Meaning |
| --- | --- |
| Key | Metadata key, for example `TemplateVersion` or `ProjectName` |
| Value | Metadata value |

Required rows:

- `TemplateVersion`
- `ProjectName`

#### Sheet: `Units`

| Column | Required | Meaning |
| --- | --- | --- |
| Enabled | Yes | `true` or `false` |
| UnitName | Yes | Unit identifier passed to generator methods |
| Description | No | Human-readable unit description |
| Area | No | Optional grouping or plant area |
| Comment | No | Free review note |

#### Sheet: `Devices`

### Code Architecture Proposal

Add a small project-definition layer to the app project, not to `TIA_LIB` initially:

```text
etu-process-generator-TiAv20/
  ProjectDefinition/
    ProjectDefinition.cs
    ProjectDefinitionLoader.cs
    ProjectDefinitionValidator.cs
    ProjectDefinitionApplier.cs
```

Responsibilities:

- `ProjectDefinition`: DTOs for metadata, units, devices, TPs, RPs, and phases.
- `ProjectDefinitionLoader`: reads the workbook using GemBox and converts rows into DTOs.
- `ProjectDefinitionValidator`: validates required fields, enum values, duplicate rows, numeric ranges, and method-specific constraints.
- `ProjectDefinitionApplier`: converts DTOs into existing `Add...` method calls on `PlcProject`.

### Execution Flow Proposal

Long-term preferred flow:

```text
load workbook
validate workbook
connect to TIA Portal
apply definition through Add... methods
CreateTags()
Upload()
CheckTags()
GenerateSiVarc()
UserInterface()
```

Important lifecycle note:

- `PlcProject()` currently connects to TIA Portal and compiles PLC in its constructor.
- For strong validation safety, TIA connection should eventually become explicit and occur after workbook validation.
- If that refactor is too large for the first pass, still add validation before `Upload()` and avoid changing runtime behavior beyond input loading.

### Validation Rules To Start With

- Required sheets must exist.
- Required columns must exist by exact header name.
- `Enabled` values must parse as booleans.
- `DeviceType` must be one of the supported values.
- Enabled rows must have required fields for their type.
- Unit names referenced by enabled device/phase rows must exist or be auto-created by documented policy.
- Duplicate `(UnitName, DeviceType, Name)` rows should be rejected or reported.
- Duplicate TP/RP numbers should be rejected.
- Numeric values should parse culture-independently; support `.` and `,` decimal separators deliberately.
- Text-list names should be required when `Type = true`.
- Unknown non-empty columns should be reported as warnings so template drift is visible.

### Migration Plan

1. Create a sample workbook template under `Templates\`.
2. Add DTOs and loader for `Metadata`, `Units`, and `Devices`.
3. Add validator and readable row/column error messages.
4. Add an application setting or command-line argument for the workbook path.
5. Convert current BL170 hardcoded rows into the template.
6. Replace hardcoded active device calls in `Project.cs` with `ProjectDefinitionApplier`.
7. Add `TechnicalParameters`, `RecipeParameters`, and `Phases` support.
8. Add a validation-only mode that does not connect to TIA Portal.
9. Refactor `PlcProject` lifecycle so validation can happen before any TIA Portal connection.

### Verification Ideas

- The current BL170 workbook produces the same active `Add...` calls as the current hardcoded `Project.cs`.
- Missing required columns fail with clear sheet and column names.
- Invalid row values fail with sheet name, row number, column name, and bad value.
- Disabled rows are ignored.
- Decimal comma and decimal dot values parse predictably.
- Build succeeds without requiring a workbook at compile time.
- Validation-only mode runs without connecting to or mutating TIA Portal.

### Risks And Open Questions

- Excel is convenient, but template drift is likely unless headers and template version are enforced.
- Culture-specific parsing can silently corrupt ranges if not handled carefully.
- The current constructor side effects make early validation harder than it should be.
- Some generated HMI/SiVArc behavior may rely on naming patterns not obvious from `Project.cs`.
- The right policy for missing units needs a decision: require explicit `Units` rows, or auto-create units from device rows.
- The template needs ownership: engineering-friendly enough to fill in, strict enough to generate safely.
